# Недостающий функционал для сценариев ПМИ (G.6, G.15, G.16, G.37, G.38)

Документ описывает, что нужно реализовать в **fx-deal-manager**, чтобы пройти автоматизированные тесты из `tests/test_pmi_appendix_g.py`, помеченные как `@pytest.mark.skip`.

**Источники:** приложение Г ПМИ FX-АСУБАНК (вер. 1.1), ТЗ FR-006, FR-010, FR-021.

**Текущее состояние:** сценарии G.5 (атомарное сохранение сделки и платежей при успешной валидации) и G.35/G.36 (операционный отчёт по сделкам, FR-019) уже покрыты. Ниже — только пробелы.

|----------|---------------|--------------|-----------------|--------------|

## G.6 — Откат сохранения при ошибке записи платежа (FR-010)

### Сценарий ПМИ

- **Входные данные:** корректная сделка USD/RUB; имитация сбоя при сохранении второго платежа.
- **Ожидаемый результат:** частично сохранённые данные не остаются в реестре; операция и платежи либо сохраняются целиком, либо не сохраняются.

### Что реализовано

В `DealService.validate_deal` операция и платежи записываются последовательно в одной SQLAlchemy-сессии с единым `commit()` в конце (`replace_payments` → `save_existing` → аудит → `commit`). При необработанном исключении до `commit` транзакция откатывается на уровне СУБД, но:

- нет явного `rollback()` в обработчике ошибок;
- нет контролируемого способа воспроизвести сбой для испытаний (ПМИ требует «тестовый режим имитации ошибки»);
- G.5 проверяет только happy path, G.6 — негативный.

### Что нужно добавить

1. **Явная транзакционная граница** в `validate_deal` (и при необходимости в `create_deal`):
   - обернуть сохранение операции + платежей + аудита в `try/except`;
   - при любой ошибке персистентности вызывать `session.rollback()` и возвращать `500`/`422` с понятным сообщением;
   - не менять `validation_status` на `VALID`, если commit не выполнен.

2. **Тестовый переключатель сбоя** (только для стенда/тестов):
   - переменная окружения, например `PAYMENT_SAVE_FAULT_AFTER=1` (номер платежа, после которого бросать исключение);
   - или заголовок `X-Test-Fault: payment-save` при `DEBUG=true`;
   - реализация — hook в `DealRepository.replace_payments` или отдельный `FaultInjectionRepository` для pytest.

3. **Интеграционный тест G.6** (после реализации):
   - создать сделку, включить fault injection;
   - вызвать `POST /deals/{id}/validate` → ожидать ошибку;
   - проверить: в БД нет платежей, `validation_status != VALID`, сделка остаётся в `DRAFT`.

### Критерий приёмки (ТЗ FR-010)

> В БД создана запись операции и связанных платежей; **целостность соблюдена**.

## G.15 — Проверка допустимых параметров сделки (FR-006)

### Сценарий ПМИ

- **Входные данные:** сделка USD/RUB, сумма в пределах лимита, активный контрагент.
- **Ожидаемый результат:** сделка проходит контроль и может быть передана на следующий этап.

### Что реализовано

`ValidationService` проверяет FLC (FR-004), валюты в НСИ, активность контрагента (FR-005). **Проверки лимитов по объёму, валюте и контрагенту нет.**

### Что нужно добавить

1. **Справочник лимитов** (НСИ / бизнес-правила):
   - таблица, например `deal_limit`:
     - `counterparty_id` (nullable — лимит по банку или глобальный);
     - `currency_code` (nullable — лимит по валюте или по всем);
     - `max_amount` (Decimal);
     - `is_active` (bool);
   - миграция Alembic + сидинг тестовых лимитов (например, VTBR/USD — 10 000 000).

2. **Сервис `LimitCheckService`** (или расширение `ValidationService`):
   - метод `check(deal) -> list[ValidationIssue]`;
   - правила минимум:
     - сумма сделки ≤ лимита для пары (контрагент, валюта списания/зачисления);
     - при отсутствии подходящего правила — пропуск или запрет (зафиксировать в ТЗ/конфиге).

3. **Вызов проверки** в `DealService.validate_deal` (до расчёта платежей) и/или в `ApprovalService.submit_deal`:
   - при нарушении — `ValidationFailedError` с полем `amount` или `limit`;
   - статус валидации `INVALID`;
   - запись в аудит `VALIDATE_FAILED`.

4. **Уведомление трейдеру** (ПМИ G.16 ссылается на то же):
   - минимум: текст ошибки в ответе API `422`;
   - опционально: запись в `notifications` / audit для UI.

### Критерий приёмки (ТЗ FR-006)

> При нарушении правил сделка блокируется; уведомление трейдеру.

### Тест G.15 (после реализации)

- сделка USD/RUB 100 000 при лимите 10 000 000 → `validate` и `submit` успешны.

## G.16 — Блокировка сделки при нарушении бизнес-правил (FR-006)

### Сценарий ПМИ

- **Входные данные:** сумма выше установленного лимита.
- **Ожидаемый результат:** передача на подтверждение заблокирована; трейдер получает сообщение о нарушении правил.

### Что нужно добавить

Тот же контур, что для G.15, с негативным сценарием:

1. При `amount > max_amount` для контрагента/валюты:
   - `POST /deals/{id}/validate` → `422`, detail содержит «limit» / «лимит»;
   - `POST /deals/{id}/submit` → `422`, если валидация была пропущена или лимит проверяется повторно на submit;
   - статус сделки остаётся `DRAFT`, `validation_status = INVALID`.

2. **Дополнительные бизнес-правила** (по архитектуре C4 — «Validation & Limit Service»):
   - запрет сделок с неактивным типом (FORWARD без даты уже есть);
   - при необходимости — дневной/месячный агрегированный лимит по контрагенту (отложить, если не в MVP).

### Тест G.16 (после реализации)

- создать сделку с суммой 50 000 000 при лимите 10 000 000;
- `validate` → 422;
- `submit` без успешной валидации → 422;
- статус не меняется на `WAITING_FOR_POSITIONER`.

## G.37 — Формирование регуляторного реестра с НСИ (FR-021)

### Сценарий ПМИ

- **Входные данные:** сделки за период, справочники валют, контрагентов и счетов.
- **Ожидаемый результат:** выгрузка содержит сведения по сделкам, валютам, контрагентам и счетам.

### Что реализовано

`GET /api/v1/reports/deals` (FR-019) — операционный отчёт: поля сделки + `counterparty_name`, **без**:

- реквизитов контрагента (BIC, страна);
- атрибутов валют (точность, наименование);
- платежей и ностро-счетов (номер счёта, банк-корреспондент);
- отдельного формата для регулятора.

### Что нужно добавить

1. **Новый endpoint**, например:
   ```
   GET /api/v1/reports/regulatory-deals
   ```
   Параметры: `trade_date_from`, `trade_date_to`, `format=csv|json|xml` (формат уточнить по ТЗ; в ТЗ указано «TBD: формат»).

2. **Сервис `RegulatoryReportService`**:
   - выборка сделок за период (все статусы кроме `CANCELLED` или по правилам регулятора);
   - join/enrich:
     - **сделка:** id, даты, тип, направление, суммы, курс, статус, трейдер, позиционер;
     - **контрагент:** id, name, bic, country, is_active;
     - **валюты:** code, name, decimal_places для buy/sell;
     - **платежи:** direction, amount, currency, value_date;
     - **ностро:** account_number, bank_name по `account_code` из payment.

3. **Структура выгрузки** (минимум для G.37):
   - CSV: одна строка на платёж **или** denormalized JSON с вложенными `payments[]` и `counterparty{}`;
   - заголовки явно помечают источник НСИ.

4. **Права доступа:** `AUDITOR`, `ADMIN` (как у FR-019).

### Критерий приёмки (ТЗ FR-021)

> Формируется выгрузка всех сделок с НСИ.

### Тест G.37 (после реализации)

- создать и провалидировать сделку с платежами;
- запросить regulatory report за период;
- проверить наличие полей контрагента, валют, account_number/bank_name в платежах.

## G.38 — Ошибка при неполной НСИ (FR-021)

### Сценарий ПМИ

- **Входные данные:** сделка без обязательного атрибута контрагента или счёта; в НСИ отсутствует обязательный атрибут.
- **Ожидаемый результат:** реестр не формируется **или** формируется протокол ошибки; пользователь видит, каких данных не хватает.

### Что нужно добавить

1. **Pre-flight валидация** в `RegulatoryReportService` перед генерацией файла:
   - для каждой сделки в периоде проверить:
     - контрагент существует и заполнены обязательные поля (id, name, bic);
     - валюты buy/sell есть в справочнике;
     - у каждого платежа VALID-сделки указан `account_code`, счёт активен в `nostro_account`;
   - собрать список ошибок: `{deal_id, field, message}`.

2. **Поведение API** (выбрать одно и задокументировать):
   - **Вариант A:** HTTP `422` + JSON `{ "errors": [...] }`, файл не отдаётся;
   - **Вариант B:** HTTP `200` + файл-протокол ошибок (отдельный `format=error-log`).

   Для ПМИ достаточно варианта A.

3. **Аудит:** событие `REGULATORY_REPORT_FAILED` с перечнем недостающих атрибутов.

### Тест G.38 (после реализации)

- сделка с платежом без `account_code` (или удалённый nostro из НСИ);
- запрос regulatory report → `422`;
- в `detail` указаны `deal_id` и недостающее поле.
