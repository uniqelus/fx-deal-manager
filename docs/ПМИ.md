# Программа и методика испытаний (ПМИ)

**Объект испытаний:** FX-АСУБАНК - мидл-офис система для ведения FX-сделок (TOD, TOM, Spot, Forward).

**Цель испытаний:** проверка соответствия реализованного прототипа функциональным требованиям FR-001 - FR-019 и сценариям защиты от 28.05.2026.

**Состав:** fx-deal-manager (бэкенд, PostgreSQL), identity-provider (внешний JWT-провайдер), fx-deal-manager-ui (веб-интерфейс).

---

## 1. Условия проведения

### 1.1. Аппаратное обеспечение

Локальная рабочая станция с поддержкой Docker (минимум 4 ядра CPU, 8 ГБ ОЗУ, 10 ГБ свободного дискового пространства).

### 1.2. Программное обеспечение

- Docker Engine 25+ и Docker Compose v2
- Python 3.12 (для сценариев тестирования и UI на http.server)
- Браузер на базе Chromium (Chrome/Edge) для проверки UI
- uv (менеджер пакетов Python) - для запуска unit-тестов
- httpie или curl для ручных API-проверок

### 1.3. Состав тестовых данных

Демо-пользователи (создаются скриптом `scripts/seed-demo-users.sh`):

| Email | Пароль | Роль |
|-------|--------|------|
| trader@demo.local | DemoPassword1! | TRADER |
| positioner@demo.local | DemoPassword1! | POSITIONER |
| auditor@demo.local | DemoPassword1! | AUDITOR |
| admin@example.com | change-this-local-admin-password | ADMIN |

Справочники НСИ (контрагенты, валюты, счета НОСТРО, бизнес-календарь) - загружаются миграциями Alembic и сидингом при первом запуске.

### 1.4. Порядок развёртывания

1. Запустить identity-provider: `cd identity-provider && docker compose up -d`.
2. Запустить fx-deal-manager: `cd fx-deal-manager && docker compose up -d --build`. Alembic-миграции применяются автоматически.
3. Создать демо-пользователей: `./scripts/seed-demo-users.sh`.
4. Запустить UI: `cd fx-deal-manager-ui && python3 -m http.server 5173`.

Готовность: `GET http://localhost:8000/api/v1/health` отвечает кодом 200.

## 2. Состав и порядок испытаний

Испытания проводятся в два этапа: автоматизированные (pytest, Testcontainers PostgreSQL) и ручные UAT-сценарии в браузере. Автоматизированные тесты запускаются командой `RUN_INTEGRATION_TESTS=1 uv run pytest tests/`. Ручные сценарии выполняются по очереди и фиксируются в протоколе испытаний.

### 2.1. Автоматизированные тесты

- `tests/test_health.py` - проверка endpoint /health (FR-000).
- `tests/test_auth.py` - проверка валидации JWT и ролевых ограничений (FR-001).
- `tests/test_deals.py` - unit-тесты сервиса сделок (FR-002, FR-004).
- `tests/test_deals_integration.py` - CRUD сделок через API (FR-002, FR-010, FR-016).
- `tests/test_validate_integration.py` - валидация и расчёт платежей (FR-004, FR-005, FR-007, FR-008, FR-009).
- `tests/test_deals_integration.py` и контрактный smoke compose-стека - полный lifecycle согласования, аудит, отправка платежей в POSITIONS-АСУБАНК (FR-011 - FR-014, FR-017, FR-018).
- `tests/test_stage6_cancel_reports.py` - отмена сделки и выгрузка отчётов (FR-015, FR-019).
- `tests/test_validation_services.py` - unit-тесты ValidationService / SettlementService.

Критерий приёмки: все автоматизированные тесты завершаются с кодом 0.

### 2.2. UAT-сценарии (ручные)

#### Сценарий 2.2.1. Happy path: SPOT EUR/RUB до статуса EXECUTED (FR-002 - FR-014, FR-017)

Шаги:

1. Войти в UI как trader@demo.local. Открыть "Новая сделка". Указать тип SPOT, направление BUY, валюты EUR/RUB, сумму 1 000 000, курс 98.50, контрагента SBER. Нажать "Сохранить".
2. На карточке сделки нажать "Проверить". Дождаться появления статуса валидации VALID и появления двух платежей.
3. Нажать "Отправить". Статус сделки переходит в WAITING_FOR_POSITIONER.
4. Выйти из системы. Войти как positioner@demo.local. Открыть "Очередь". Найти созданную сделку, открыть сводную форму.
5. Нажать "Согласовать". Статус становится EXECUTED. В аудите появляется событие POSITION_SEND.

Критерий приёмки: статус сделки в БД равен EXECUTED, в журнале аудита присутствуют события CREATE, VALIDATE, STATUS_CHANGE (DRAFT → WAITING), STATUS_CHANGE (WAITING → APPROVED), POSITION_SEND, STATUS_CHANGE (APPROVED → EXECUTED).

#### Сценарий 2.2.2. Возврат на исправление с комментарием (FR-013)

Шаги:

1. Создать и отправить сделку как трейдер (см. 2.2.1).
2. Войти как позиционер. Открыть сделку. Нажать "Вернуть на исправление". В модальном окне ввести комментарий "Уточните курс".
3. Войти как трейдер. Открыть сделку в статусе REJECTED. Нажать "Взять в работу". Статус возвращается в DRAFT.
4. Изменить курс. Заново выполнить шаги валидации, отправки, согласования.

Критерий приёмки: возврат без комментария блокируется ошибкой 422. После согласования сделка достигает статуса EXECUTED. Аудит содержит обе попытки.

#### Сценарий 2.2.3. Ошибка валидации - неактивный контрагент (FR-005, FR-006)

Шаги:

1. Войти как трейдер. Создать сделку с контрагентом BCSB (статус "На проверке" в справочнике).
2. Нажать "Сохранить".

Критерий приёмки: запрос возвращает 422 с сообщением о неактивном контрагенте, сделка не сохраняется.

#### Сценарий 2.2.4. Запрет на согласование собственной сделки (FR-001)

Шаги:

1. Войти как пользователь с двойной ролью TRADER + POSITIONER (или эмулировать назначение роли через PATCH /users/{id}/role в IdP).
2. Создать, валидировать и отправить сделку.
3. Перейти в очередь согласования. Попытаться нажать "Согласовать".

Критерий приёмки: попытка завершается ошибкой 403 "Self-approval is not allowed". Аудит фиксирует попытку.

#### Сценарий 2.2.5. RBAC - трейдер не имеет доступа к согласованию, позиционер не создаёт сделки (FR-001)

Шаги:

1. Под учётной записью трейдера попытаться выполнить `POST /api/v1/deals/{id}/approve` через curl.
2. Под учётной записью позиционера попытаться выполнить `POST /api/v1/deals`.

Критерий приёмки: оба запроса завершаются кодом 403.

#### Сценарий 2.2.6. Журнал аудита (FR-018)

Шаги:

1. Войти как auditor@demo.local. Открыть "Журнал аудита".
2. Задать фильтр по идентификатору сделки из сценария 2.2.1.
3. Открыть экспорт в CSV.

Критерий приёмки: журнал содержит полную хронологию изменений сделки; экспорт корректно открывается в Excel/LibreOffice.

#### Сценарий 2.2.7. Отчёт по сделкам за период (FR-019)

Шаги:

1. Войти как auditor@demo.local или admin. Выполнить запрос `GET /api/v1/reports/deals?trade_date_from=2026-05-01&trade_date_to=2026-05-31&format=csv`.
2. Сохранить ответ как deals_report.csv. Открыть в табличном редакторе.

Критерий приёмки: файл содержит заголовок и строки сделок, в столбце counterparty_id отображаются ожидаемые контрагенты, поля trade_date и value_date в формате ISO.

#### Сценарий 2.2.8. Отмена сделки в DRAFT (FR-015)

Шаги:

1. Войти как трейдер. Создать сделку в DRAFT.
2. На карточке сделки нажать "Отменить сделку", ввести комментарий, подтвердить.

Критерий приёмки: статус сделки CANCELLED, попытка отменить сделку из других статусов завершается ошибкой 403, в аудите присутствует STATUS_CHANGE с комментарием отмены.
