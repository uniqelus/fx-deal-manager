# Руководство администратора FX-АСУБАНК

Документ описывает порядок развёртывания, конфигурирования и обслуживания подсистемы обработки FX-сделок.

## 1. Архитектура и зависимости

Подсистема состоит из трёх независимых сервисов. Бэкенд fx-deal-manager предоставляет REST API на порту 8000 и хранит данные в PostgreSQL. Сервис identity-provider предоставляет авторизацию по JWT на порту 8083 и не модифицируется в рамках проекта. Веб-интерфейс fx-deal-manager-ui развёртывается как статические файлы на порту 5173 либо проксируется через nginx-контейнер из docker-compose.dev.yml. Сервис identity-provider должен быть запущен до старта fx-deal-manager, так как fx-deal-manager при старте получает JWKS-ключи для верификации токенов.

## 2. Развёртывание

Подготовьте каталоги трёх репозиториев рядом. Скопируйте `.env.example` в `.env` в каталоге fx-deal-manager и при необходимости измените параметры подключения к PostgreSQL и URL identity-provider. Запустите identity-provider командой `docker compose up -d` из его репозитория. Затем запустите fx-deal-manager командой `docker compose up -d --build`. При первом запуске Alembic применит миграции автоматически, если переменная окружения `AUTO_MIGRATE` равна `true` (значение по умолчанию). Веб-интерфейс запускается через `python3 -m http.server 5173` из каталога fx-deal-manager-ui или через профиль `docker compose -f docker-compose.yml -f docker-compose.dev.yml up -d`.

## 3. Создание учётных записей

Демонстрационные учётные записи создаются скриптом `scripts/seed-demo-users.sh`, который обращается к API identity-provider. Скрипт регистрирует трёх пользователей (TRADER, POSITIONER, AUDITOR) с предзаданными паролями и назначает им роли через bootstrap-учётную запись администратора. Для production-среды используйте `POST /api/v1/auth/register` и `PATCH /api/v1/users/{id}/role` в identity-provider напрямую через Swagger или curl.

## 4. Управление миграциями БД

Миграции Alembic хранятся в `alembic/versions/`. Создание новой миграции выполняется командой `alembic revision --autogenerate -m "описание"`; применение - `alembic upgrade head`. Откат - `alembic downgrade -1`. В контейнере используется `docker compose exec api alembic upgrade head`.

## 5. Резервное копирование

Снимок состояния PostgreSQL формируется командой `docker compose exec postgres pg_dump -U fxuser -d fxdb > backup_$(date +%F).sql`. Восстановление - `docker compose exec -T postgres psql -U fxuser -d fxdb < backup.sql`. Журнал аудита нельзя редактировать в обход API, поэтому отдельной выгрузки не требуется.

## 6. Мониторинг

Endpoint `GET /api/v1/health` возвращает статус подключения к БД и версию приложения. Логи доступны через `docker compose logs -f api`. В режиме отладки можно установить `LOG_LEVEL=DEBUG` в `.env`. Все обращения к API логируются middleware RequestLoggingMiddleware с указанием correlation_id, метода, пути и кода ответа.

## 7. Интеграция с системой позиций

Передача согласованных сделок в систему позиций реализована через адаптер `PositionSystemAdapter`. В режиме прототипа используется in-process stub `InProcessPositionStub`, накапливающий записи в памяти. Для подключения реальной системы реализуйте альтернативный адаптер, удовлетворяющий контракту из `src/fx_deal_manager/integrations/position_stub.py`, и зарегистрируйте его в `ApprovalService.__init__`.

## 8. Управление ролями и доступом

Роли назначаются в identity-provider. Поддерживаются роли TRADER, POSITIONER, AUDITOR и ADMIN. Маппинг роли на доступные операции зашит в декораторе `require_role` в `src/fx_deal_manager/api/dependencies.py`. Для добавления новых ролей расширьте allowlist в декораторах endpoint-ов и обновите UI guards в `js/app.js`. Изменение ролей пользователя не требует перезапуска сервиса; токены обновляются при следующем запросе refresh.

## 9. Аудит и отчётность

Журнал аудита фиксируется автоматически на каждом изменении состояния сделки и редактировании справочников. Записи хранятся в таблице `audit_log_entry` и доступны через `GET /api/v1/audit-events`. Сводные отчёты доступны через `GET /api/v1/reports/deals` с фильтрами по периоду, статусу, типу сделки и контрагенту; формат вывода - JSON или CSV.

## 10. Завершение работы и удаление окружения

Остановка контейнеров - `docker compose down`. Полное удаление с томами и БД - `docker compose down -v`. После удаления тома база данных будет пересоздана при следующем запуске.
